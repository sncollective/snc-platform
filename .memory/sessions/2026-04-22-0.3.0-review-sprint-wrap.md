## Arc

1. **[mini-player-stream-end-spinner](../active/stories/mini-player-stream-end-spinner.md)** — bug confirmed first: killed a live ffmpeg-pushed Maya stream and watched the mini-player spin forever. First fix attempted used Vidstack's `useMediaState("error")` — never fired because hls.js retries live manifests indefinitely and the error state never populates. Switched to server-authoritative polling of `/api/streaming/status` in `GlobalPlayer`. First version of that caused a regression on playout mini-players because `/live` sets `streamType: "live"` on playout channels too (they both use HLS live semantics client-side) — my poll check `type !== "live"` dismissed any playout mini-player on first tick. Re-scoped to `media.contentType === "live"` (the actual live-vs-playout differentiator) and added a 3-consecutive-miss threshold (30s grace) so transient unpublish/republish flaps don't kick viewers out. Re-verified: Maya's stream ended → mini-player dismissed after ~30s; playout mini-player survives reload cycles.

2. **[live-page-coming-soon-flash](../active/stories/live-page-coming-soon-flash.md)** — existing `!isStreaming && !isLoading` gate didn't prevent the flash because `isStreaming` is derived from `selectedChannel`, which is null on first render (before the default-channel auto-select effect fires post-paint). Gate rewritten to `!hasChannels && !isLoading` — "Coming Soon" means "no active channels at all," matching the truth SSR hands over immediately.

3. **[notification-inbox](../active/features/notification-inbox.md)** — seeded demo notifications for Pat (topped up to 12 so the 400px-max-height dropdown scroll could be verified). Two fix-in-flight findings:
   - Dropdown bounced on scroll. Ark UI Popover defaults to `strategy: "absolute"` — positions relative to document, while the trigger is inside a sticky nav (viewport coords). Floating-ui's autoUpdate lagged the recompute. Added `strategy: "fixed"` to the positioning prop.
   - Dropdown rendered behind the nav bar. Zag-JS applies `z-index: var(--z-index)` as an *inline style* on the positioner; inline beats our `.positioner { z-index: var(--z-popover) }` class rule. The menu module already had this solved with `!important` on the direct property; applied the same pattern to `popover.module.css`. Fix propagates to all Popover usages, not just notification-bell.

4. **[patron-badges](../active/features/patron-badges.md)** — needed a subscription fixture strategy the feature spec explicitly noted was deferred. Inserted two fake active rows in `user_subscriptions` for Pat (S/NC Monthly platform + Maya Chen Monthly creator) with fake Stripe IDs — `resolveUserBadges` only reads local subscription state, no Stripe round-trip. Verified both-badges rendering in Maya's channel chat, platform-only in other channels, historical correctness via direct DB inspection of `chat_messages.badges`. One design-iteration request parked for 0.3.1 ([patron-badges-icon-variant](../backlog/patron-badges-icon-variant.md)).

5. **[set-up-snc-login-prompt](../active/features/set-up-snc-login-prompt.md)** — all seeded users have `credential` providers, so the banner wouldn't naturally appear for anyone. Temporarily flipped Pat's `accounts.provider_id` from `credential` to `google` (preserving the row + password hash for restore), had the user sign in with password first to establish a session, then swap, then re-verify. Banner appeared, dismiss + localStorage persistence worked, clearing the key re-showed the banner, "Send verification code" triggered the OTP flow and delivered to Mailpit. Restored Pat's credential at end of session.

6. **Three passes with minimal fix work** — `live-page-auto-scroll` was vague ("investigated and addressed") and already implicit in TanStack Router's `scrollRestoration: true` — accepted as-is. `error-page-hydration-pop-in` already had the `isMounted` + `suppressHydrationWarning` pattern in place, verified by hitting an unknown route and confirming no flicker/warning. `content-detail-video-thumbnail-overlap` already had the `.coverArt { max-width: 100%; height: auto }` constraint, verified visually.

## Learnings

- **Event-state signals for live streams need to be server-authoritative, not client-inferred.** Vidstack's `useMediaState("error")` looked like the obvious signal for stream-end but doesn't fire for HLS 404s because hls.js retries forever on live manifests. The `/api/streaming/status` endpoint is the actual source of truth — whatever feature needs to know "is this live stream still running," poll status, don't infer from media-state. Same pattern will likely apply to future stream-state UX (reconnect UI, stream-quality fallback, etc.).

- **`streamType` and `contentType` both carry "live" signals but mean different things.** `/live` sets `streamType: "live"` on all channels because HLS live streaming is the *transport*. `contentType` distinguishes between live creator streams (live) and playout schedules (playout). Business logic that treats "live" as a class of creator broadcasts should gate on `contentType`, not `streamType`. Burned a fix cycle on this.

- **Zag-JS inline styles beat CSS class rules.** The Popover positioner's inline `z-index: var(--z-index)` (with default `auto`) silently overrode `.positioner { z-index: var(--z-popover) }`. The menu module had this lesson already — should have grepped for the `!important` comment before writing the popover positioner class the first time. Same Zag gotcha will bite any future Ark UI component that uses popper positioning (tooltip, combobox, hover-card, etc.). Worth a scan-accessibility / scan-stylistic rule addition if it recurs.

- **Grace windows matter for server-authoritative dismiss logic.** First poll-and-dismiss attempt fired on single-poll channel-missing signal, which kicked users off during normal stream reload flaps. 3 × 10s = 30s grace is conservative but well past any transient unpublish/republish window SRS produces. When a dismiss action is user-visible and destructive (losing playback state), err toward a grace window that covers normal operational variance.

- **Dev-data fixtures unblock review for Stripe-gated features.** `patron-badges` had a "defer E2E testing until subscription fixtures exist" note in the design matter. Turns out direct `INSERT` into `user_subscriptions` with fake Stripe IDs is perfectly adequate for review because `resolveUserBadges` reads local state — the Stripe side is only relevant at subscription-creation time. Pattern is probably reusable for future Stripe-gated features whose reads are local-only.

- **Provider-flip harness for OAuth-only UX.** The `set-up-snc-login-prompt` banner only appears for users without a `credential` provider, but all seeded users have one. Flipping `provider_id` on an existing account row (preserving the password hash) lets a user sign in via credential to establish a session, then simulate OAuth-only state for the UX test, then restore. Worth generalizing into a dev tooling helper if this pattern shows up again — e.g., `scripts/platform/simulate-oauth-user.sh <email> <provider>`.
