# Session: four tier-2 quick-wins scoped and shipped to review the day before Animal Future

Started with the active tier looking overcrowded (~60 `done` items not yet swept into a release) and a planned full `refactor-scan` sweep. Cleaned up refactor intent-only items from active, then scanned the 159-item backlog for 0.3.0-shipping-adjacent quick wins, scoped four into active, and implemented all four to `stage: review`. Went long on one of them after grounding surfaced that the scoped design was wrong — revised the story in place before implementing.

## Arc

**Refactor-scan prep.** Audited the 14 `implementing` + 2 `drafting` refactor items in active. Classified as 0 landed / 14 partial / 2 intent-only initially; two of the "partial" turned out to be intent-only on closer read (`refactor-bookings-desktop-mobile-dedup` — no target files identified; `refactor-error-extraction-helper` — motivating pattern largely gone via earlier typed-`AppError` migration). Moved four intent-only items to backlog so the scan has a clean slate, left the 12 partials at `implementing` for scan findings to merge into rather than duplicate.

**Backlog scan for quick wins.** Parallel explore over `platform/.memory/backlog/*.md`, cross-referenced against 0.3.0 scope. Produced a 6-way split: 2 gate-blocker CI tests (routed to `/e2e-triage` at gate time, not scoped here) + 4 tier-2 polish items. Scoped all four via `/scope` — promoted from backlog to `.memory/active/stories/<slug>.md` with expanded frontmatter. Parked a new follow-up: `channels-human-readable-url-slugs` surfaced during the mini-player scope, deferred to post-0.3.0 since it's a schema change + backfill across all channel URL surfaces, not a mini-player local fix.

**Implementation — all four via `/implement` sequentially.**

1. **`notification-preferences-ui-nav-link`** — nav surface turned out to be the user-menu dropdown, not a settings sub-nav (settings pages don't use `ContextShell` — that pattern is admin/governance/creator-manage only per `platform-0010`). One-line config add to `apps/web/src/config/auth-menu-items.ts` + `Bell` icon import + extended the existing user-menu test.

2. **`mini-player-go-to-content-loses-channel`** — `validateSearch` on the `/live` route with `zod/mini`, `Route.useSearch()` seeds `selectedChannelId` (URL param wins over `defaultChannelId`), `MediaMetadata.contentUrl` now carries the channel id. Added a test asserting URL-param priority; the initial assertion `queryByText(/42 viewers/).toBeNull()` failed because the non-selected channel still appears in the combobox option list — tightened to "URL channel appears *more* times than default" (hero + option vs just option).

3. **`chat-sent-message-double-render`** — paused before implementing. Grounding via Explore surfaced that the scoped design was **wrong**: no client-side optimistic insert exists today, so the "optimistic + echo duplication" premise didn't match the code. Diagnosed actual cause: React strict-mode double-mount + reconnect-loop in `chat-context.tsx:381-389` creates orphan WS connections that all server-side-join the same room; one server broadcast hits multiple ws.onmessage handlers in the same tab. Revised the story matter in place per the live-kind-file convention, committed the revision with the user's approval before implementing. Fix: module-level `WeakSet<WebSocket>` for aborted sockets + identity check on close. Chose WeakSet over the story's documented `__abortReconnect` property-augmentation (cleaner typing, no casts); noted the deviation in "What shipped." Single file, no protocol/server/type changes.

4. **`theater-mode-tab-bar-hide`** — `AppShell` in `routes/__root.tsx` already guarded `<NavBar>` with `!isTheater` but rendered `<BottomTabBar />` unconditionally. Wrapped it in the same guard. Extended the existing `t` keyboard effect in `live.tsx` to also handle Escape (only when theater is already active — no regression outside theater). Two small edits, no new tests added (the NavBar theater pattern is itself untested, kept consistent).

Full web suite green throughout: 1599 → 1600 tests (one test added in mini-player).

## Learnings

- **"Root cause" in a story.md at scope time is a hypothesis, not a finding.** `chat-sent-message-double-render` was scoped as a story with a clear-looking root cause + two fix options. `/implement`'s Step 3 grounding revealed both were wrong — the premise didn't match the code. A feature would have gone through `/design` first, which spawns Explore agents specifically to verify design assumptions; the mis-diagnosis would have surfaced there instead. Sizing heuristic to carry forward: if a story's matter says "likely" or "probably" about the root cause, OR offers "more robust" fix options requiring new protocol fields / shared-type changes, size up to feature and run `/design`.

- **`/implement` Step 3 grounding is the final line of defense.** It caught the chat mis-diagnosis before any code was written. The skill's rule — "don't send agents into stale design" — worked. Worth internalizing as "read the file the story names modifying before writing a single edit."

- **WeakSet > property augmentation for per-instance flags on built-ins.** The story's documented `__abortReconnect` on WebSocket would have required `as WebSocket & { __abortReconnect?: boolean }` casts at every call site. `new WeakSet<WebSocket>()` at module scope — no cast, no type pollution, auto-GC when the socket is collected. Design intent preserved; implementation detail improved.

- **Tier 1 / tier 2 quick-wins split works.** Separating "gate blockers that will go in via gate pipelines anyway" (CI test-drift items flowing through `/e2e-triage`) from "optional scope pulled now via `/scope`" (the four polish items) stopped me from bundling testing items into the scope-now flow. Route items to their natural pipeline instead of overriding.
