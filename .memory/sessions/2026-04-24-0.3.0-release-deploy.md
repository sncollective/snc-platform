# Session: /release-deploy 0.3.0 — all four gates cleared, fix-in-flight + batch-tracker pattern at scale

Ran the full release-deploy sequence for 0.3.0 (Animal Future event-day release) in one session. Opened with `/review` on 5 items carrying over from yesterday, then `/release-deploy 0.3.0` through the refactor → security → testing → documentation gate sequence. 13 total fix-in-flight blockers written + fixed across the four gates; the rest parked as batch-tracker backlog items rather than individual stories. 79 items archived into `.memory/releases/0.3.0/`. ## Arc

**Pre-gate review.** `/review` on the remaining 4 stories (chat double-render, mini-player channel loss, notification nav link, theater tab-bar) + the resumable-uploads-tus feature. The feature review surfaced real bugs mid-test — a 400MB upload went in fine but processing failed with ENOENT. Root cause: tusd was writing to Garage under `tus/<opaque-id>`, not the canonical `content/<id>/media/<filename>` path the probe/transcode pipeline expected, and `getFileExtension` on a slash-containing key returned the whole key as the extension (the "bin" fallback didn't trip because `"tus/xyz".split(".").pop()` yields the full string, not undefined). Fix-in-flight: `tusd-hooks.routes.ts:handlePostFinish` now S3-copies `tus/<id>` to the canonical key, deletes the source + `.info` sidecar, then runs `completeUploadFlow` with the canonical path (no more `skipKeyValidation`). Also fixed `getFileExtension`, pre-existing `boss.schedule` mock gap that had been failing since commit 8d826ae2, and added `filename`/`type` to the web Uppy `allowedMetaFields` so the canonical rename has a filename to work with. Verified with a 4GB HEVC round-trip: upload → rename → probe → transcode → playback. All 5 items passed and bound to 0.3.0.

**Security gate (full-codebase, 10 rules).** 5 parallel scan agents. 63 findings. 5 S1 blockers fixed inline:

3 deferred S1s parked for design work (tusd hooks network auth, playout callback timing-safe, multipart sign-part ownership). 4 root infra items split out to the parent repo (caddy CSP, curl-pipe-bash bun bootstrap, SSH keyscan TOFU, garage.toml secret templating) because tag-taxonomy routes `deploy + user-station` findings to root, not sub-project.

**Post-gate CI firefighting — Test & Build.** Demo-deploy dispatch surfaced three CI blockers in the Forgejo "Platform: Test & Build" stages, fixed as a sequence of submodule bumps. (1) The 16 pre-existing `exactOptionalPropertyTypes` typecheck errors noted as a known issue at archival time were actually in the CI path — fixed inline (`79274e6`) rather than parked since the demo needed green. (2) E2E `global.setup.ts` auth failed with a 404 whose body was the app's own SSR 404 page — POSTs to `/api/auth/sign-in/email` were hitting Vite and falling through to TanStack Start instead of reaching the API on :3100. First attempted fix (`0b5e842`) added `server.proxy` to `vite.config.ts` and looked correct but changed nothing at runtime. (3) Root cause of (2): the `nitro()` Vite plugin replaces Vite's dev server with Nitro's — Vite's `server.proxy` is silently ignored. Moved the same proxy config to Nitro's `devProxy` with H3 wildcard routes (`/api/**`, `/uploads/**`) in `2c17759`; verified against `apps/web/node_modules/nitro/dist/_chunks/dev.mjs:491` that `devProxy` routes register on Nitro's H3 app via httpxy, preserving the incoming path. Local dev is unaffected — Caddy still does the routing; `VITE_API_URL` stays unset so the block is a no-op outside CI.

**Post-gate CI firefighting — Deploy workflow.** Demo-deploy then ran end-to-end through test + typecheck + build, reached the `deploy-demo` job, and failed at `/var/run/act/workflow/1.sh: line 5: syntax error: unexpected end of file` after ssh-agent + ssh-add succeeded. `bash -n` on the YAML-rendered run block passed clean, so this was specific to how `act` compiled the step. The deploy workflow hadn't actually been exercised end-to-end since 0.2.0 — the pattern was a massive unquoted heredoc (`<<DEPLOY`) containing ~40 `set_env KEY "${SECRET_VALUE}"` calls that got expanded locally by CI bash. Two rounds of fixes: first `99d4433` added the 0.3.0 env surface (imgproxy, tus/Garage admin token, streaming, SMTP, OAuth) to both prod + demo `set_env` blocks plus the YAML-level `env:` bindings — this is what introduced the scale that tripped act. Then `33f5941` rewrote the shape entirely: build `/tmp/env-patch` locally on CI with `printf '%q'`-quoted values, `scp` to remote, use a **quoted** heredoc (`<<'DEPLOY'`) on the remote that merges the patch file via a `while IFS='=' read -r key _rest; sed -i "/^${key}=/d"; done; cat patch >> .env` loop. Eliminates the entire local-expansion attack surface and the fragile nested-quoting. Both prod + demo workflows migrated to the new shape.

**E2E post-proxy iterations + content-failure parking.** After the Nitro devProxy fix, the next e2e run surfaced a Playwright version mismatch — `@playwright/test` semver `^1.52.0` in `apps/e2e/package.json` had drifted to `1.59.1` via bun.lock, but the workflow's Playwright docker image was still pinned to `v1.52.0-noble`, so chromium refused to launch ("Executable doesn't exist at .../chromium_headless_shell-1217"). Bumped to `v1.59.1-noble` (`f6acb28`). Then the `Upload test report` step failed — `actions/upload-artifact@v3` was deprecated by GitHub in early 2025 and Forgejo follows GitHub's action lifecycle; bumped to `@v4` (`a79cd84`). The run after that actually exercised all 113 tests: ~20 real content/UI failures landed (strict-mode `getByText('Maya Chen')` ambiguities, empty-state conditional renders on `/landing`/`/admin/playout`/`/live`, post-register email-verification branch with no Mailpit in CI services). E2E doesn't gate either deploy workflow (`needs:` omits `test-e2e`), so parked the whole batch as `e2e-ci-first-run-failures-2026-04-24` (submodule `80281d2`) grouped by root-cause shape (selector brittleness / empty-state cascade / auth-flow email-verification) for post-event `/scope` rather than drive-by fixing pre-event.

**Demo host prereqs.** Deploy workflow shape was correct but hit two remote-side misconfigurations on first successful SSH-through: `error: unzip is required to install bun` (the deploy's bun-bootstrap fallback path needed `apt-get install -y unzip` on the Debian host) and `sudo: a terminal is required to read the password` on `sudo systemctl restart snc-api snc-web` (non-interactive SSH has no TTY, so any sudo prompt fails). ## Learnings

- **Batch-tracker backlog items are a legit output shape for gate scans.** A full-codebase refactor scan produced 246 findings across 7 libraries. Writing each as an individual item would drown the board in low-value noise; writing zero would lose the scan's findings entirely. The middle path: write a dated batch-tracker (`refactor-scan-2026-04-24-findings.md`) with findings grouped by existing-feature-to-expand / new-work-needed / accessibility / SEO / etc., plus explicit "how to consume this" + "revisit if" sections so `/scope` or a future scan can mine it without re-doing the triage. Applied the same pattern to security (63 findings), testing (~10), and documentation (~7). Four batch-trackers in `backlog/`, ~15 bound stories in `releases/0.3.0/stories/`.

- **Fix-in-flight scales to a 5-10-blocker gate batch if the fixes are mechanical.** `/implement` ceremony per story would have been wasteful for fixes like "delete 5-line duplicate useEffect" or "add `requireRole('stakeholder')`" or "escape three strings in three templates." Writing the story first (for the audit trail) and then applying the fix inline in one pass, re-running the test suite, flipping stage-done — ~5-10 minutes per fix. The `/review` skill already documents this as the fix-in-flight escape hatch; the release-deploy gate fix sequence is the same shape at bigger batch size. Worth codifying in `item-pipelines.md` that fix-in-flight is a gate-scale move, not just a review-one-item move.

- **Tool-permission edges on sub-agents are not always deterministic.** One of seven parallel refactor-scan sub-agents reported `Bash` and `Skill` denied mid-run; the other six used Bash fine. Re-launching with explicit "use Glob + Read + Grep; don't rely on Bash `find`" in the prompt succeeded. The failure mode is the sub-agent giving up and reporting "I can't do the job" rather than retrying with available tools. Phrasing the prompt to name the specific tools to use (`Glob pattern="..."` as an example) avoids the dead-end. Worth updating scan-skill prompt templates to always spell out the tool choice.

- **S3 test harness lesson.** `vi.fn().mockImplementation(() => ({...}))` doesn't work cleanly as a constructor — `new Upload(...)` with that mock fails at runtime (`uploader.done is not a function`). Class-based mock (`class MockUpload { constructor(options) { ... } async done() { ... } }`) works. Had to debug via transient `console.log` in the test to see the wrapped error (`wrapS3Error` obscures the underlying `TypeError`). Pattern: when a mock needs constructor semantics, use a `class`, not a `vi.fn` factory.

- **Adjacent-bug audits during a fix-in-flight need scope discipline.** While fixing `invite.ts` to use `getFrontendBaseUrl()`, I grepped for all `BETTER_AUTH_URL` uses and found 5 more call sites with the same latent bug class (streaming-connect redirects, go-live notification URL, mastodon-auth post-callback redirect). Temptation: expand the fix. Right call: park the 5 sites as a separate story and ask the user before expanding. User chose park → single new backlog item (`refactor-better-auth-url-frontend-redirects`). Keeps the fix narrow and the audit trail honest.

- **The docs gate bar is legitimately lower than the other three.** Missing docs rarely block a release — a reviewer can read the code. The bar that actually matters: does an existing doc *actively mislead*? Only `content.md`'s upload-paths section qualified (described a code path that would be stale post-deploy). Everything else was absent-but-not-misleading. The gate-finding vs. park split works out to 1 bound story + a batch-tracker for everything else. Worth encoding in `/docs-triage` that "absent" ≠ "blocking" for event-scoped releases.

- **Release archival works as flat-file rename, not folder move.** The skill's step 8 says `git mv .memory/active/<slug>/ .memory/releases/<version>/<slug>/` assuming folder-per-item, but this codebase uses flat files (`stories/<slug>.md`, `features/<slug>.md`). Substituting `git mv .memory/active/<kind>/<slug>.md .memory/releases/<version>/<kind>/<slug>.md` preserves the kind grouping. git-untracked items need plain `mv` as a second pass. Worth updating the skill wording to cover both layouts.

- **Vite `server.proxy` is silently ignored under `nitro/vite`.** The `nitro()` plugin replaces the Vite dev server wholesale, so top-level `server.proxy` config in `vite.config.ts` has no effect in TanStack Start projects using Nitro. Use `nitro.devProxy` instead, with H3 wildcard route patterns (`/api/**`, not `/api`). Failure mode is quiet — config looks right, code-review passes, the proxy block never runs. General lesson: when adjusting any dev-server axis (proxy, middleware, rewrites, CORS, headers) in this codebase, check which plugin owns the server before reaching for the Vite-native knob. Verified against Nitro's source at `apps/web/node_modules/nitro/dist/_chunks/dev.mjs:491` — `devProxy` routes register on the H3 app via httpxy and preserve the incoming path. Revisit if we drop Nitro or TanStack Start moves off the nitro/vite integration.

- **Unquoted heredocs with many variable expansions are fragile under `act`.** `bash -n` on the rendered YAML run block passed clean, but act still failed with `line 5: syntax error: unexpected end of file` when the heredoc body had ~40 `set_env KEY "${VALUE}"` calls. The pattern `ssh host bash <<DELIMITER ... ${SECRET} ... DELIMITER` relies on local bash to expand `${SECRET}` before streaming to stdin; at scale this becomes a combinatorial quoting minefield where any secret with a special character could break the outer shell's parsing, and act's compilation to `/var/run/act/workflow/N.sh` amplifies edge cases beyond what local `bash -n` catches. **Pattern to use instead**: build a config-patch file locally with `printf '%q'` quoting, `scp` to remote, then use a *quoted* heredoc (`<<'DELIMITER'`) on the remote — no local expansion, no nested-quote hazards, and `printf %q` handles every shell-special character correctly by construction. Bonus: the remote merge loop (`while IFS='=' read -r key _rest; sed; done < patch; cat patch >> .env`) is easier to read than 40 shape-identical function calls. Pay the one-time file-transfer cost once; get quote-safety for free forever. Revisit if: act adopts a more capable YAML→shell compiler, or we switch CI runners entirely.

- **Pinned container image + semver package range = silent drift.** `@playwright/test` at `^1.52.0` in `package.json` resolved through `bun.lock` to `1.59.1` on any fresh install; the workflow's Playwright docker image stayed pinned to `v1.52.0-noble`, so the runtime had a 1.59 client trying to drive a 1.52 browser set. Failure mode is clear (chromium "Executable doesn't exist"), but the drift happens whenever the lockfile regenerates, with no signal until the next CI run. **Bound values must move in lockstep**: either pin both tight (`~1.52.0` + image `v1.52`) or bump both together as routine maintenance. Applies beyond Playwright — same class for any container image that bundles a client library version (cypress, puppeteer, database drivers pinned to server-major). Revisit if: we adopt a lint that cross-checks package.json/lockfile resolved versions against container image tags in workflow YAML.

- **Deploy-host prereqs: two items that aren't worth leaving to implicit discovery.** Non-interactive SSH has no TTY, so any `sudo` command in the deploy script must have a matching `NOPASSWD` rule in `/etc/sudoers.d/` — scope by exact command path (`NOPASSWD: /bin/systemctl restart snc-api`), not broad `ALL`. And any on-host bootstrap fallback that uses `curl | bash` inherits that tool's unstated deps — bun needs `unzip` to unpack its zip archive, which isn't present on clean Debian/Alpine hosts. Both are discoverable only when the deploy actually runs end-to-end, and both block the entire release. Worth codifying a **first-boot prereqs checklist** per deploy target kind so the next fresh demo / staging / prod host doesn't re-hit these. Revisit if: deploy-runbook promotion makes this material first-class, or we adopt containerized deploys where host prereqs are baked in.
