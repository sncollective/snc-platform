# Session: 0.3.1 post-launch triage — 3 prod bugs, 3 surfaced gaps, 1 deploy flake

Opened 0.3.1 as a post-launch triage bundle for bugs found after 0.3.0 went live for the Animal Future show on 2026-04-24. Three fixes scoped, implemented, tested, committed, and (by session end) running in prod with all three destinations live. Three follow-up gaps surfaced during triage but not fixed — candidates for 0.3.2. One deploy-workflow flake hit once and cleared on retry.

## Arc

**Calendar event PATCH dropped visibility.** Creator tried saving a show event as `public`; it persisted as `internal`. The PATCH handler at `creator-events.routes.ts` builds an `updates` object field-by-field and was missing the `visibility` assignment — Zod accepted it, form sent it, handler dropped it. On re-fetch the column's `notNull().default("internal")` filled in the blank, reading exactly like "public flipped to internal." Paired with the 0.3.0 client-side fix ("event-form no longer overwrites visibility to public for existing show-type events on edit") — that fix made the form send the right value; this one makes the server persist it. Regression test asserts `.set()` receives `visibility: "public"` on PATCH; fixture updated to match schema (was missing `visibility` column).

**on_forward classifier misrouted creator publishes.** Creator configured and activated Twitch + YouTube simulcast, stopped and restarted the stream, neither external platform showed live. SRS logged `response={"code":0,"data":{"urls":[]}}` — the API returned zero URLs, not even Liquidsoap. Root cause: `on_publish`'s `ensureLiveChannelWithChat(creatorId, sessionId, body.stream)` creates a `channels` row with `srs_stream_name = <creator's publish name>, is_active = true` on every creator go-live. The subsequent `on_forward` callback then sees its own row via `isPlayoutStream()`, classifies the publish as "playout," and takes the early-return branch that omits both the Liquidsoap URL and the creator's simulcast destinations. Every creator go-live since 0.3.0 shipped silently broken — it just wasn't exercised until today. Fix: invert the branch order. Session lookup by `srsClientId` first (creator publishes carry a validated-stream-key session row; Liquidsoap publishes don't). Session match → creator branch (Liquidsoap + creator simulcast). No session → fall through to `isPlayoutStream` for Liquidsoap-originated playout publishes. Added structured log per branch (`on_forward_creator` / `on_forward_playout` / `on_forward_unknown`) so the next incident on this path reads in one log scan rather than a re-investigation.

**Stream keys + simulcast services bypassed the shared permission model.** Admin hit `/creators/<id>/manage/streaming` for a creator they didn't own and got "only creator can manage." The services had hardcoded `requireOwner()` helpers querying `creator_members` for `role === "owner"` directly — bypassing `checkCreatorPermission`, which already has the admin bypass used by every other creator-scoped write path (creator-events, creator-profile, etc.). Owner-only shipped before the shared permission flow landed and never got migrated. Fix: added `manageStreaming` to `CREATOR_ROLE_PERMISSIONS` (owner: true, editor: false, viewer: false — preserves current role semantics; editor doesn't gain streaming as a side effect). Replaced per-service `requireOwner` with `checkCreatorPermission(userId, creatorId, "manageStreaming", userRoles)`. Threaded `getRoles(c)` through the 7 creator-scoped route handlers. Web side needed no change — the manage/streaming route already computes `isOwner = isAdmin || memberRole === "owner"`; API rejection was what surfaced the error.

**Deploy + prod verification.** Three commits on platform/main (b175c99 calendar, a6136f6 streaming, a58855e memory bundle). First `/release-deploy 0.3.1` attempt failed at the deploy-prod job's `apk add openssh-client curl` step with a TLS error fetching `dl-cdn.alpinelinux.org`. Transient — cleared on retry. Deploy completed. Prod verification walk surfaced two downstream issues: (a) Twitch simulcast still failing with `errno=32 (EPIPE)` then `errno=104 (ECONNRESET)` after the on_forward fix landed — SRS forwarding correctly, Twitch rejecting the push; the stored stream key had gone stale since OAuth captured it; (b) YouTube ran clean on first try. Rotating the Twitch stream key in the Twitch dashboard didn't propagate (see gap 1 below); manual `UPDATE simulcast_destinations SET stream_key = '...'` on the prod DB plus a stream restart cleared it. Final state: creator stream reaches S/NC + Liquidsoap playout + Twitch + YouTube.

## Learnings worth preserving

**Session identity is truth for on_forward classification.** Where the system has proof of publisher identity (validated stream-key session row), that should dominate name-based classification. Name-based classifiers are fragile to collision, especially when the system itself can synthesize new rows sharing the name. Pattern generalizes beyond on_forward — any branching where a stream name or similar weak identifier is checked against a shared table should look for a stronger co-existing identity signal first.

**The `channels` table conflates two concepts.** Genuine platform playout channels (S/NC TV, S/NC Classics — Liquidsoap-driven, long-lived, admin-configured) vs. per-creator live-takeover rows (auto-created at go-live, ephemeral, creator-scoped). Both use `is_active = true` and a `srs_stream_name`. The classifier layer works around the ambiguity via session lookup, but the schema is the deeper design gap. A `kind` column (`playout` vs `live-takeover`) would make the intent explicit and retire `isPlayoutStream`'s name-based guessing entirely. Low-priority structural cleanup — parked mentally, not yet written up.

**Admin-not-threaded-through is a silent failure pattern.** Services that predate the shared permission flow keep shipping with their own hardcoded ownership checks. Today's was stream-keys + simulcast; could easily exist elsewhere. Worth a proactive scan — any `creator_members` direct query that isn't `checkCreatorPermission` is suspect. Especially worth running before the co-op opens creator-team self-service broadly, where admins will need to act on behalf of creators who run into onboarding issues. Refactor-gate scan rule candidate.

**Missing success-path logs cost triage time.** The on_forward handler had logging for rejection but nothing on successful-forward. First 10 minutes of today's triage were spent proving SRS was forwarding correctly and that the API was returning data — visible only through SRS's own log of the callback response body. Added structured success logs (branch + urlCount + creatorId) during the fix; the next incident will read the log rather than reconstruct state. General posture: on any decision path where external diagnostics depend on the outcome, log the outcome with enough structure to identify the branch taken.

**OAuth-captured credentials go stale.** Stream keys (Twitch especially — rotatable via dashboard), and eventually OAuth refresh tokens (YouTube) will drift from what's stored. Today required manual DB surgery because `createCreatorSimulcastDestination` doesn't dedupe by `(creator_id, platform)` — reconnecting through OAuth silently no-ops on duplicate. Gap 1 in the open-threads list.

## Open threads / follow-ups parked

**Gap 1: OAuth reconnect should update existing destinations, not silently no-op.** `streaming-connect.routes.ts` catches "destination may already exist" and leaves the stale row. Upsert-on-`(creator_id, platform)` in `createCreatorSimulcastDestination` (or a new `upsertCreatorSimulcastDestination`) + a visible "Re-authenticate / Refresh key" button on the manage/streaming UI. Small service fix (~50 LOC) + one UI affordance. Candidate for 0.3.2.

**Gap 2: Deploy workflow's `apk add` is a flake risk.** `platform-deploy-prod.yml` and `platform-deploy-demo.yml` both `apk add --no-cache openssh-client curl` inside `alpine:latest` on each deploy. Transient TLS/CDN hiccups on dl-cdn.alpinelinux.org will block deploys when they happen. Two robust changes: pin `alpine:3.22` (not `latest`) and bake openssh-client + curl into a purpose-built deploy base image cached in the registry. Eliminates the runtime apk entirely. Deploy-infra + root-scope tag since this touches workflow YAMLs at root, not platform source.

**Gap 3: Schema disambiguation for `channels` playout vs live-takeover.** `isPlayoutStream` currently classifies by name; session-first classifier works around the ambiguity but the underlying conflation is still there. Adding a `kind: playout | live-takeover` column and tightening `isPlayoutStream` to `kind = 'playout'` would make the intent explicit. Not required for any specific failure today, but reduces the fragility of the area. Low priority.

**Verification threads.** All three 0.3.1 stories are at `stage: review`, bound to release 0.3.1. They've been prod-verified end-to-end (visibility persists; creator stream simulcasts to both destinations; admin can manage stream keys and simulcast for creators they don't own). Not yet flipped to `stage: done` — that happens via `/review` on each, which also re-asserts the release binding and prompts for any residual prod-only caveats to go into the release file's Prod verification section. Next session step.

**Release closeout.** Platform commits unpushed at session end. `/release-deploy 0.3.1` not yet run — it would run the four gates, archive the three stories from `active/` to `releases/0.3.1/`, and flip release status `planned → released`.
