# Widen playout_queue to a source-polymorphic schema

Unit 1 of `unified-channel-model-creator-content-playable`. Lands first — it's the regression gate
the other three units build on.

## Scope
Make `playout_queue` carry either a playout item or a creator content row, mirroring the
already-polymorphic `channel_content` (`playout-queue.schema.ts:24-27`).

**File**: `apps/api/src/db/schema/playout-queue.schema.ts` + a migration generated via
`drizzle-kit generate` — NEVER hand-write the SQL/journal/snapshot (`.claude/rules/drizzle-migrations.md`).

```ts
playoutItemId: text("playout_item_id").references(() => playoutItems.id, { onDelete: "cascade" }), // now nullable
contentId: text("content_id").references(() => content.id, { onDelete: "cascade" }),               // new, nullable
// table extras:
check("playout_queue_one_source", sql`num_nonnulls(playout_item_id, content_id) = 1`),
```

Workflow: edit schema → `bun run --filter @snc/api db:generate` → `bun run --filter @snc/api db:migrate`.

## Acceptance
- [ ] Migration generated by drizzle-kit (random two-word name, real timestamp, snapshot present) and applied clean.
- [ ] All pre-existing `playout_queue` rows satisfy the CHECK (they all set `playout_item_id`) — verify before/after.
- [ ] A row with both source columns null, or both set, is rejected by the DB CHECK.
- [ ] Admin playout queue behavior unchanged (admin rows set `playout_item_id`, `content_id` null).

## Notes
The URI-resolution layer (`resolveContentUri`, `playout-orchestrator.ts:88-112`) already resolves
from either source — this story only unblocks the queue table to hold the content source.
